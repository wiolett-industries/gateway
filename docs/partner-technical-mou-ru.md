# Техническое описание продукта Gateway

Этот документ описывает Gateway как технический продукт для партнерского технического меморандума (MOU): назначение, архитектурную модель, текущие возможности и планируемые направления развития. Текст рассчитан на технического партнера, которому важно понять продукт без погружения в детали внутренней реализации.

Статусы внутри списка возможностей используются только для ясности:

- **Доступно**: возможность уже присутствует в продукте.
- **Планируется**: возможность относится к плану развития и требует новой реализации.
- **Есть техническая база**: в продукте уже есть часть архитектуры или смежные рабочие процессы, но полноценная возможность требует отдельного расширения.

Возможности со статусами **Планируется** и **Есть техническая база** отражают направление развития продукта. Они не фиксируют сроки, обязательный состав поставки или коммерческие условия; такие условия могут определяться отдельными юридическими или проектными документами.

## 1. Краткое описание продукта

Gateway — панель управления инфраструктурой, разворачиваемая в собственной инфраструктуре пользователя. Он объединяет в одном интерфейсе управление входящим HTTP/HTTPS-трафиком через nginx, Docker-контейнерами и приложениями, сертификатами, доменами, базами данных, логированием, мониторингом, статус-страницами, автоматизацией и операциями с участием ИИ.

Gateway предназначен для небольших и средних инфраструктурных команд, которым нужен единый безопасный инструмент для повседневных операций. Обычно такие операции распределены между nginx-конфигурациями, Docker-хостами, скриптами командной строки, клиентами баз данных, хранилищами сертификатов, панелями мониторинга и системами оповещений.

Ключевая идея продукта — дать операторам и интеграциям контролируемый доступ к инфраструктурным операциям без необходимости выдавать прямой root-доступ к каждому серверу. Управляемые хосты подключаются к Gateway через легкие серверные агенты, а пользователи работают через интерфейс, API, OAuth, MCP, вебхуки и ИИ-ассистента в пределах своих прав доступа.

## 2. Архитектурная модель

Gateway разворачивается как центральный сервер управления. Управляемые хосты запускают отдельные серверные агенты и подключаются к Gateway исходящим соединением по gRPC, то есть по протоколу взаимодействия сервисов. Соединение защищается mTLS: Gateway и управляемый сервер проверяют друг друга по сертификатам. В этой модели Gateway не открывает админский порт на каждом сервере, а принимает защищенные подключения от серверных агентов.

Текущая модель включает:

- центральное веб-приложение и API Gateway;
- PostgreSQL для продуктовых данных;
- Redis для сессий, кеша и ограничения частоты запросов;
- опциональный ClickHouse для структурированного логирования;
- gRPC-слой управления для подключений серверных агентов;
- `nginx-daemon` для nginx-серверов, которые принимают HTTP/HTTPS-трафик и направляют его дальше к нужным приложениям;
- `docker-daemon` для Docker-хостов, то есть серверов, где запускаются контейнеры и приложения;
- `monitoring-daemon` для серверов мониторинга, которые отдают только метрики без права управлять nginx или Docker.

Управляемым серверам не нужны входящие порты для администрирования. Сервисы продолжают работать при недоступности Gateway: существующие nginx-конфигурации обслуживают трафик, Docker-контейнеры продолжают выполняться, а серверные агенты переподключаются после восстановления центрального сервера.

## 3. Единый список возможностей продукта

### 3.1 Панель управления, разворачиваемая в собственной инфраструктуре

**Статус: Доступно**

Gateway предоставляет единый веб-интерфейс и API для управления инфраструктурой: входящим HTTP/HTTPS-трафиком, Docker-контейнерами и приложениями, сертификатами, базами данных, логами, мониторингом, статус-страницами, оповещениями и автоматизацией. Продукт ориентирован на эксплуатационные задачи, а не только на просмотр состояния или редактирование конфигураций.

### 3.2 Управление серверами через агентскую модель

**Статус: Доступно**

Gateway управляет хостами через фоновые серверные агенты. Сервер регистрируется через одноразовый токен подключения, после чего использует mTLS-идентификацию по сертификату: сервер доказывает Gateway, что он именно тот управляемый хост, за который себя выдает. Текущие типы агентов включают nginx, Docker и мониторинг. Gateway отслеживает доступность серверов, статус подключения и работоспособности, совместимость версий, логи, метрики и обновления агентов.

### 3.3 Маршрутизация HTTP/HTTPS-трафика через nginx

**Статус: Доступно**

Gateway управляет nginx на нескольких серверах. В этой роли nginx работает как входной прокси: принимает внешний HTTP/HTTPS-трафик и направляет запросы к нужному внутреннему сервису. Поддерживаются прокси-хосты, редиректы, 404-заглушки, папки, порядок хостов, WebSocket, пользовательские заголовки, правила переписывания запросов, терминация SSL/TLS (расшифровка трафика на стороне nginx), списки доступа, шаблоны nginx, проверки работоспособности, логи, статистика, а также режимы полного управления и интеграции с уже существующей nginx-конфигурацией.

### 3.4 Управление Docker-контейнерами и приложениями

**Статус: Доступно**

Gateway управляет Docker-серверами, контейнерами и приложениями: образами, реестрами, томами, сетями, задачами, логами, консолями, просмотром и редактированием файлов, секретами, переменными окружения, метками, портами, политиками перезапуска, ресурсными ограничениями и сценариями очистки. Для пользователя это означает, что базовые операции с контейнерами доступны из Gateway без прямого входа на сервер.

Отдельно поддерживаются определения развертываний, слоты развертывания, публикация новой версии, откат к предыдущей версии, операции остановки и переключения, состояние развертываний, CI/CD-вебхуки, сценарии загрузки и пересоздания образов, а также зашифрованные секреты развертываний. В практическом смысле это дает управляемый способ обновлять приложение, откатываться на предыдущую версию и запускать обновление из внешней системы сборки.

### 3.5 Управление подключениями к базам данных

**Статус: Доступно**

Gateway позволяет хранить и управлять подключениями к PostgreSQL и Redis с зашифрованными учетными данными. Поддерживаются проверки подключений, история состояния, просмотр схем, таблиц и строк PostgreSQL, сканирование ключей Redis, SQL/Redis-консоли с учетом прав доступа, операции записи и отдельные права для выполнения запросов и раскрытия учетных данных.

### 3.6 SSL-сертификаты и внутренняя PKI

**Статус: Доступно**

Gateway включает ACME SSL, загруженные сертификаты и внутреннюю PKI. PKI здесь означает внутреннюю инфраструктуру сертификатов: Gateway может создавать собственные центры сертификации и выпускать сертификаты для внутренних сервисов. Поддерживаются сертификаты Let's Encrypt, HTTP-01 и DNS-01 проверки владения доменом, расписания обновления, корневые и промежуточные центры сертификации, шаблоны сертификатов, серверные и клиентские TLS-сертификаты, сертификаты для подписи кода и email, списки отозванных сертификатов (CRL), экспорт сертификатов и привязка сертификатов к прокси-хостам.

Материал приватных ключей хранится в зашифрованном виде, а операции экспорта и раскрытия защищены отдельными правами доступа.

### 3.7 Реестр доменов и DNS-проверки

**Статус: Доступно**

Gateway содержит центральный реестр доменов. Система отслеживает DNS-записи, использование доменов в прокси-хостах и SSL-сертификатах, плановые DNS-проверки и DNS-статус в интерфейсе. Поддерживаются проверки DNS-записей типов A, AAAA, CNAME, CAA, MX и TXT.

### 3.8 Структурированное логирование

**Статус: Доступно**

Gateway может принимать логи внешних сервисов в ClickHouse. Поддерживаются среды логирования, схемы данных, настройки хранения, токены, которые позволяют только отправлять логи, API для одиночной и пакетной отправки логов, проверка уровня важности, ограничения частоты запросов, частичное принятие пакетных запросов, поисковый интерфейс и официальный TypeScript SDK.

Логирование является опциональной возможностью: если ClickHouse не настроен, остальной продукт продолжает работать.

### 3.9 Мониторинг и контроль состояния

**Статус: Доступно**

Gateway собирает метрики CPU, памяти, диска и сети, состояние сервисов, статус подключения и работоспособности серверных агентов, логи серверов и статус обновлений. Сервер мониторинга позволяет получать метрики хоста без предоставления Gateway прав на управление nginx или Docker.

### 3.10 Статус-страницы и оповещения

**Статус: Доступно**

Gateway поддерживает статус-страницы, состояние сервисов, инциденты, правила алертов, вебхуки оповещений и журналы доставки. Эти функции позволяют использовать Gateway не только как панель управления, но и как инструмент оповещения об инцидентах и информирования о них.

### 3.11 Программный доступ и автоматизация

**Статус: Доступно**

Gateway поддерживает токены REST API, OAuth 2.0 Authorization Code + PKCE, удаленную точку подключения MCP и токены для отправки логов. PKCE — это способ безопасного OAuth-входа для публичных клиентов, например CLI. MCP (Model Context Protocol) используется для контролируемого подключения внешних ИИ/MCP-клиентов к возможностям Gateway. Программный доступ разделен по типам токенов и OAuth-токенам доступа, привязанным к конкретному ресурсу. API/OAuth-права ограничены фактическими правами владельца токена.

Вебхуки Docker-развертываний уже позволяют запускать загрузку образа, пересоздание контейнера и развертывание из внешних CI/CD систем.

### 3.12 Опциональный ИИ-ассистент

**Статус: Доступно**

Gateway включает опционального ИИ-ассистента, который может использовать провайдера, совместимого с OpenAI API, инструменты Gateway, операции с проверкой прав, уточняющие вопросы, согласования для опасных действий, базу знаний, сохраненные диалоги и пользовательские настройки доступа к инструментам. ИИ-ассистент отключен по умолчанию и не отправляет данные внешнему провайдеру до явного включения администратором.

### 3.13 Администрирование, ролевая модель доступа и аудит

**Статус: Доступно**

Gateway поддерживает OIDC-аутентификацию, то есть корпоративный вход через внешний сервис идентификации, пользователей, группы, детальные права доступа, ограничения прав на уровне отдельных ресурсов, выданные API/OAuth/MCP-разрешения, журнал аудита, состояние первичной настройки, управление обновлениями и отображение лицензионного статуса. Чувствительные операции, такие как раскрытие секретов, экспорт сертификатов, раскрытие учетных данных баз данных, редактирование подключенных Docker-томов, доступ к консолям и доступ к исходной конфигурации, защищены отдельными правами.

### 3.14 Подписанные обновления

**Статус: Доступно**

Gateway и серверные агенты поддерживают подписанные описания релизов и проверку контрольной суммы SHA256. Проще говоря, Gateway проверяет подпись и контрольную сумму обновления и не применяет его при несовпадении. Это снижает риск установки неподтвержденных образов Gateway или бинарных файлов серверных агентов.

### 3.15 Лицензионный статус продукта

**Статус: Доступно**

Gateway имеет лицензионные статусы Community, Homelab и Enterprise. Текущие уровни продукта отображаются в приложении и позволяют активировать лицензионные ключи, но на текущем этапе не блокируют функциональность по уровню тарифа.

### 3.16 Консольный клиент

**Статус: Планируется**

CLI предполагается развивать как консольный клиент для взаимодействия с Gateway: просмотр и управление ресурсами, серверами, прокси-хостами, Docker-контейнерами и приложениями, базами данных, развертываниями, токенами и сценариями автоматизации. CLI будет опираться на существующую API/OAuth-базу и проектируется как инструмент, удобный и для операторов, и для CI/CD сценариев.

### 3.17 Подключение доменов через Cloudflare

**Статус: Планируется**

Cloudflare-интеграция призвана дать администраторам возможность подключать аккаунт или токен Cloudflare, импортировать зоны и управлять DNS-записями из Gateway. Основной фокус первой версии: DNS-зоны и DNS-записи для доменов, прокси-хостов и SSL-сценариев.

### 3.18 Управляемый сервер баз данных

**Статус: Планируется**

Это направление предполагает отдельный серверный агент для управления базами данных. Цель — дать возможность создавать и управлять экземплярами PostgreSQL, Redis и потенциально MySQL прямо из раздела Databases без необходимости вручную создавать отдельный Docker-контейнер и затем подключать его как внешнюю базу данных.

### 3.19 Bastion-сервер для SSH-доступа

**Статус: Есть техническая база**

Bastion-сервер в перспективе станет контролируемой точкой входа по SSH. Это позволит управлять доступом на основе пользователей и групп, вести аудит сессий, выдавать временный доступ и открывать безопасный терминал через Gateway. В текущей модели уже существует техническая база для такого сценария, но полноценный bastion-агент и сценарии управления SSH требуют отдельной реализации.

### 3.20 Worker-сервер и Gateway CI/CD

**Статус: Планируется**

Worker-сервер предназначен для выполнения задач, скриптов и пайплайнов на стороне Gateway. Это единая возможность для встроенного CI/CD внутри Gateway: запуск по событиям, запуск по расписанию, секреты, логи, подтверждения, история задач и автоматизация развертываний. Worker-серверы расширят Gateway от панели управления инфраструктурой к легкому исполнителю операционных задач.

### 3.21 Шлюз доступа к ИИ-провайдерам

**Статус: Планируется**

Шлюз доступа к ИИ-провайдерам призван позволить администраторам подключать ключи внешних ИИ-провайдеров, например провайдеров, совместимых с OpenAI API, а пользователям — создавать внутренние ключи Gateway. Пользователи смогут обращаться к ИИ через Gateway, а Gateway будет отвечать за контроль доступа, выбор внешнего провайдера, учет использования, лимиты и аудит.

### 3.22 Миграции контейнеров и прокси-хостов

**Статус: Планируется**

Миграции призваны позволить переносить контейнеры, развертывания и прокси-хосты между серверами. Цель — перенос между серверами с минимальным простоем, а там, где это технически возможно, без простоя. Если схема нагрузки не позволяет гарантировать перенос без простоя, Gateway предполагается использовать для контролируемого запасного сценария, проверки результата и подсказок для отката.

### 3.23 Просмотр и управление S3/MinIO

**Статус: Планируется**

S3/MinIO-модуль предполагается развивать как менеджер объектного хранилища, то есть хранилища файлов и объектов, совместимого с S3 API. Возможность включает подключение S3-совместимых хранилищ, просмотр бакетов и объектов, загрузку и скачивание файлов, базовые операции с объектами, управление учетными данными и базовые сценарии прав доступа.

### 3.24 Упрощенный ИИ-режим / интерфейс Gateway через чат

**Статус: Планируется**

Упрощенный ИИ-режим (AI Lite mode) предполагается развивать как отдельный интерфейс через чат для пользователей без опыта в DevOps. В этом режиме пользователь работает не через полный административный интерфейс, а через ассистента, который ведет его через операции, задает уточняющие вопросы, заполняет формы и выполняет действия только в рамках разрешенных прав доступа и согласований.

### 3.25 Отдельное Studio-окно

**Статус: Есть техническая база**

Studio в перспективе станет отдельным рабочим окном в стиле JupyterLab: дерево файлов, вкладки с терминалами, консоли, логи и редакторы для серверов и контейнеров. В текущем продукте уже существуют отдельные всплывающие консоли, логи и файловые менеджеры; Studio объединит эти инструменты в более цельное рабочее пространство.

### 3.26 Адаптивный операторский интерфейс

**Статус: Есть техническая база**

Адаптивный интерфейс в перспективе сделает Gateway удобным на телефонах и планшетах для мониторинга, подтверждений, быстрых действий и реакции на инциденты. В текущем интерфейсе уже есть адаптивная база, но требуется отдельная работа по проверке и полировке мобильных сценариев.

## 4. Безопасность и доступ

Gateway проектируется как привилегированная панель управления инфраструктурой, поэтому безопасность является частью базовой архитектуры, а не дополнительным модулем.

Ключевые принципы:

- вход через OIDC SSO;
- отсутствие встроенной базы паролей в текущей версии;
- подключения серверных агентов только изнутри наружу;
- mTLS-идентификация для управляемых серверов;
- одноразовые токены подключения, которые нужны только для первичного подключения сервера;
- зашифрованное хранение секретов и приватных ключей;
- детальные права доступа, ограниченные конкретными ресурсами;
- отдельные права на раскрытие и экспорт для чувствительных операций;
- журнал аудита для административных, автоматизированных и ИИ-операций;
- проверка подписанных обновлений для Gateway и серверных агентов.

Эта модель проектируется так, чтобы опасные инфраструктурные операции были видимыми, аутентифицированными, ограниченными правами доступа и проверяемыми через аудит.
